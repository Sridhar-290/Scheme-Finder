---
name: API type shapes
description: Actual fields on UserProfile and SavedScheme — AI subagents often assume more fields exist than are in the OpenAPI spec.
---

UserProfile (from generated api.schemas.ts):
  id, name, state, age, income?, occupation?, gender?, category?, createdAt?
  — NO maritalStatus, bplStatus, or education fields

UserProfileInput (update body):
  name?, state?, age?, income?, occupation?, gender?, category?

SavedScheme:
  id, schemeId, scheme: Scheme, notes?, savedAt
  — uses savedAt (not createdAt)

SchemeResult (eligibility result item):
  scheme, reason, matchScore, requiredDocuments[], missingCriteria?
  — requiredDocuments is on SchemeResult, NOT on the nested scheme object

**Why:** The OpenAPI spec was written before the profile was extended. The generated types are the source of truth. Subagents writing profile forms or eligibility result renderers consistently add fields that don't exist.

**How to apply:** Always grep the generated api.schemas.ts before writing form fields or property accesses against these types.
